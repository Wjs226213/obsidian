## 摘要

我们推出 DeepSeek-V3.2，这是一个将高计算效率与卓越推理及智能体（Agent）性能相协调的模型。DeepSeek-V3.2 的关键技术突破如下：

1. **DeepSeek 稀疏注意力（DSA）**：我们引入了 DSA，一种高效的注意力机制，在长上下文场景下大幅降低计算复杂度的同时保持模型性能。
2. **可扩展强化学习框架**：通过实施稳健的强化学习协议并扩展后训练计算，DeepSeek-V3.2 的表现可与 GPT-5 相媲美。值得注意的是，我们的高计算变体 DeepSeek-V3.2-Speciale 超越了 GPT-5，推理能力与 Gemini-3.0-Pro 相当，在 2025 年国际数学奥林匹克（IMO）和国际信息学奥林匹克（IOI）中均取得金牌成绩。
3. **大规模智能体任务合成流水线**：为将推理能力集成到工具使用场景中，我们开发了一种新颖的合成流水线，能够系统性地大规模生成训练数据。该方法促进了可扩展的智能体后训练，在复杂交互环境中的泛化能力和指令跟随鲁棒性方面取得了显著提升。

## 1. 引言

推理模型的发布（DeepSeek-AI, 2025; OpenAI, 2024a）标志着大语言模型（LLM）演进的关键时刻，推动了可验证领域整体性能的大幅提升。自这一里程碑以来，LLM 的能力迅速提升。然而，近几个月来出现了一个明显的分歧：开源社区（MiniMax, 2025; MoonShot, 2025; ZhiPu-AI, 2025）持续取得进展，但闭源专有模型（Anthropic, 2025b; DeepMind, 2025a; OpenAI, 2025）的性能轨迹以显著更陡峭的速度加速。因此，闭源和开源模型之间的性能差距不仅没有收敛，反而在扩大，专有系统在复杂任务中展现出越来越优越的能力。

通过分析，我们识别出限制开源模型在复杂任务中能力的**三个关键缺陷**：

1. **架构层面**：对传统注意力（vanilla attention）机制的过度依赖严重制约了长序列的效率，这对可扩展部署和有效后训练构成了重大障碍。
2. **资源分配层面**：开源模型在后训练阶段的计算投入不足，限制了其在困难任务上的表现。
3. **智能体层面**：开源模型在泛化和指令跟随能力方面明显落后于闭源对应产品（EvalSys, 2025; Li et al., 2025; Luo et al., 2025），阻碍了其在实际部署中的有效性。

为解决这些关键限制，我们采取了以下措施：

- 首先，引入 **DSA**，一种高效的注意力机制，显著降低计算复杂度，有效解决效率瓶颈。
- 其次，开发了**稳定可扩展的 RL 协议**，允许在后训练阶段进行大规模计算扩展。该框架分配的后训练计算预算超过预训练成本的 10%，释放了高级能力。
- 第三，提出了一种新颖的流水线，以促进工具使用场景中的可泛化推理。首先利用 DeepSeek-V3（DeepSeek-AI, 2024）方法实施冷启动阶段，在单一轨迹中统一推理和工具使用。随后推进大规模智能体任务合成了生成超过 1,800 个不同环境和 85,000 个复杂提示。这些广泛的合成数据驱动 RL 流程，显著增强了模型在智能体上下文中的泛化和指令跟随能力。

DeepSeek-V3.2 在多个推理基准上取得了与 Kimi-k2-thinking 和 GPT-5 相当的性能。此外，DeepSeek-V3.2 显著推进了开源模型的智能体能力，在 EvalSys (2025)、Li et al. (2025)、Luo et al. (2025) 引入的长尾智能体任务上表现出色。DeepSeek-V3.2 成为智能体场景中极具成本效益的替代方案，显著缩小了开源与前沿专有模型之间的性能差距，同时产生更低的成本。

值得注意的是，为推动开源模型在推理领域的边界，我们放宽了长度约束以开发 DeepSeek-V3.2-Speciale。该模型在 IOI 2025、ICPC 世界总决赛 2025、IMO 2025 和 CMO 2025 中均取得了金牌成绩。

## 2. DeepSeek-V3.2 架构

### 2.1. DeepSeek 稀疏注意力（DSA）

DeepSeek-V3.2 使用与 DeepSeek-V3.2-Exp 完全相同的架构。与 DeepSeek-V3.1-Terminus 相比，DeepSeek-V3.2 唯一的架构修改是通过持续训练引入了 DeepSeek 稀疏注意力（DSA）。

#### DSA 原型

DSA 原型主要由两个组件组成：**闪电索引器（Lightning Indexer）** 和 **细粒度 Token 选择机制**。

- **闪电索引器**计算查询 token h_t 与前驱 token h_s 之间的索引分数 I_{t,s}，决定查询 token 应选择哪些 token。由于闪电索引器的头数较少且可以用 FP8 实现，其计算效率非常出色。
- **细粒度 Token 选择机制**仅检索对应 Top-k 索引分数的键值条目，然后通过在查询 token 和稀疏选中的键值条目之间应用注意力机制来计算注意力输出。

#### 在 MLA 下实例化 DSA

考虑到从 DeepSeek-V3.1-Terminus 进行持续训练，我们在 MLA 基础上实例化 DSA。在内核层面，每个键值条目必须在多个查询之间共享以保证计算效率。因此，我们在 MLA 的 MQA 模式下实现 DSA，其中每个潜在向量（MLA 的键值条目）将在查询 token 的所有查询头之间共享。

### 2.1.1. 持续预训练

从 DeepSeek-V3.1-Terminus 的基础检查点（其上下文长度已扩展到 128K）出发，我们执行持续预训练和后训练以创建 DeepSeek-V3.2。

#### 密集预热阶段

我们首先使用短暂的预热阶段初始化闪电索引器。在此阶段，保持密集注意力并冻结除闪电索引器外的所有模型参数。为使索引器输出与主注意力分布对齐，使用 KL 散度损失作为索引器的训练目标。预热使用 10^-3 的学习率，仅训练 1000 步，每步包含 16 个 128K token 的序列，总计 2.1B token。

#### 稀疏训练阶段

索引器预热后，引入细粒度 token 选择机制并优化所有模型参数以适应 DSA 的稀疏模式。在此阶段使用 7.3×10^-6 的学习率，为每个查询 token 选择 2048 个键值 token。训练主模型和索引器各 15,000 步，每步包含 480 个 128K token 的序列，总计 943.7B token。

### 2.2. 等价性评估

- **标准基准测试**：DeepSeek-V3.2-Exp 在长序列上显著提高了计算效率，同时在短上下文和长上下文任务上均未观察到明显的性能下降。
- **人类偏好**：使用 ChatbotArena 作为间接评估框架，DeepSeek-V3.1-Terminus 和 DeepSeek-V3.2-Exp 的 Elo 评分非常接近。
- **长上下文评估**：在 AA-LCR 和 Fiction.liveBench 评估中，DeepSeek-V3.2-Exp 一致优于 DeepSeek-V3.1-Terminus。

### 2.3. 推理成本

DSA 将主模型的核心注意力复杂度从 O(L²) 降低到 O(Lk)，其中 k（≪L）是选中的 token 数量。结合优化实现，DSA 在长上下文场景中实现了显著的端到端加速。推理成本随 token 位置的变化显示，DeepSeek-V3.2 在预填充和解码阶段均比 DeepSeek-V3.1-Terminus 具有更低的成本。

## 3. 后训练

持续预训练后，我们执行后训练以创建最终的 DeepSeek-V3.2。后训练同样使用稀疏注意力。

### 专家蒸馏（Specialist Distillation）

对于每个任务，我们首先开发一个专门针对该特定领域的专业模型，所有专家模型均从同一预训练检查点微调。框架涵盖六个专业领域：**数学、编程、通用逻辑推理、通用智能体任务、智能体编码和智能体搜索**，所有领域均支持思考和非思考模式。每个专家使用大规模强化学习进行训练。一旦专家模型准备就绪，就用于生成最终检查点的领域特定数据。

### 混合 RL 训练（Mixed RL Training）

DeepSeek-V3.2 采用 **GRPO（Group Relative Policy Optimization）** 作为 RL 训练算法。将推理、智能体和人类对齐训练合并到一个 RL 阶段中，有效平衡了不同领域的性能，同时避免了多阶段训练范式常见的灾难性遗忘问题。

#### 扩展 GRPO 的稳定性策略

1. **无偏 KL 估计**：修正 K3 估计器，使用重要性采样比获得无偏 KL 估计，消除了系统性估计误差。
2. **离策略序列掩码**：引入二进制掩码到 GRPO 损失中，屏蔽导致显著策略偏离的负序列，提高训练稳定性。
3. **保留路由（Keep Routing）**：保留推理框架中的专家路由路径，训练时强制使用相同的路由路径，确保优化相同的专家参数。
4. **保留采样掩码（Keep Sampling Mask）**：保留 top-p/top-k 采样的截断掩码并应用于训练，确保两个策略共享相同的行为子空间。

### 3.2. 工具使用中的思考

#### 3.2.1. 思考上下文管理

我们观察到，复制 DeepSeek-R1 的策略（在第二轮消息到达时丢弃推理内容）会导致显著的 token 浪费。为此，我们开发了专门为工具调用场景设计的上下文管理策略：

- 历史推理内容**仅在引入新的用户消息时**丢弃。如果仅附加工具相关消息（如工具输出），推理内容在整个交互过程中**保留**。
- 当推理痕迹被移除时，**工具调用及其结果的历史**仍保留在上下文中。

#### 3.2.2. 冷启动

利用推理数据（非智能体）和非推理智能体数据，通过精心设计的提示将两种能力整合。模型被要求在推理过程中执行工具调用，使用特殊的 `<think></think>` 标签标记推理路径。

#### 3.2.3. 大规模智能体任务

我们使用的智能体任务包括：

| 任务类型 | 任务数量 | 环境类型 | 提示来源 |
|---------|---------|---------|---------|
| 代码智能体 | 24,667 | 真实 | 提取 |
| 搜索智能体 | 50,275 | 真实 | 合成 |
| 通用智能体 | 4,417 | 合成 | 合成 |
| 代码解释器 | 5,908 | 真实 | 提取 |

- **搜索智能体**：使用基于 DeepSeek-V3.2 的多智能体流水线生成多样化高质量训练数据。
- **代码智能体**：通过挖掘 GitHub 上数百万 issue-PR 对构建大规模可执行环境，涵盖 Python、Java、JavaScript、TypeScript、C、C++、Go 和 PHP 等多种语言。
- **代码解释器智能体**：利用 Jupyter Notebook 作为代码解释器处理复杂数学、逻辑和数据科学推理任务。
- **通用智能体**：使用自动环境合成智能体合成了 1,827 个面向任务的环境。这些任务难以求解但易于验证，例如旅行规划任务。

## 4. 评估

### 4.1. 主要结果

我们在多个基准上评估模型，包括 MMLU-Pro、GPQA Diamond、HLE、LiveCodeBench、Codeforces、AIME 2025、HMMT、Terminal Bench 2.0、SWE-Verified、BrowseComp、τ2-bench、MCP-Universe、MCP-Mark 和 Tool-Decathlon。

- **推理能力**：DeepSeek-V3.2 在推理任务上取得了与 GPT-5-High 相当的性能，但略逊于 Gemini-3.0-Pro。与 K2-Thinking 相比，以更少的输出 token 取得了可比的分数。
- **代码智能体**：在 SWE-bench Verified 和 Terminal Bench 2.0 上显著优于开源 LLM。
- **搜索智能体**：使用标准商业搜索 API 评估，通过上下文管理方法取得了优异成绩。
- **工具使用**：大幅缩小了开源和闭源 LLM 之间的性能差距。

### 4.3. 合成智能体任务的消融实验

实验表明：
- 合成任务对 DeepSeek-V3.2-Exp（12%准确率）和前沿闭源模型（最高62%准确率）均具有足够挑战性。
- 在合成数据上进行 RL 可以有效泛化到不同任务和真实世界环境（Tau2Bench、MCP-Mark、MCP-Universe）。
- 仅限制在代码和搜索场景中的 RL 无法改善这些基准的性能，进一步突显了合成数据的潜力。

### 4.4. 搜索智能体的上下文管理

为解决 128K 上下文窗口的限制，我们引入了上下文管理策略，在 token 使用超过上下文窗口长度的 80% 时启用：

1. **Summary（摘要）**：总结溢出的轨迹并重新开始 rollout（平均步数从140增加到364，性能从53.4提升到60.2）。
2. **Discard-75%**：丢弃轨迹中前 75% 的工具调用历史以释放空间。
3. **Discard-all**：通过丢弃所有先前的工具调用历史重置上下文（简单但高效，得分 67.6）。

在不同计算预算下，上下文管理通过允许模型扩展测试时计算量，提供了更多空间执行额外步骤，带来显著的性能提升。

## 5. 结论、局限性与未来工作

### 主要贡献

1. 引入 **DSA**，解决了关键计算复杂度问题而不牺牲长上下文性能。
2. 通过增加计算预算，DeepSeek-V3.2 在推理基准上取得了与 GPT-5 相当的性能。
3. 大规模智能体任务合成流水线显著增强了工具使用能力，为开源 LLM 的可靠、可泛化 AI 智能体解锁了新的可能性。
4. 高计算变体 DeepSeek-V3.2-Speciale 在 IMO 和 IOI 中取得金牌，为开源 LLM 设立了里程碑。

### 局限性

1. **知识广度**：由于总训练 FLOPs 较少，DeepSeek-V3.2 的世界知识广度仍落后于领先的专有模型。计划在未来的迭代中通过扩展预训练计算来弥补。
2. **Token 效率**：DeepSeek-V3.2 通常需要更长的生成轨迹（更多 token）才能匹配 Gemini-3.0-Pro 等模型的输出质量。
3. **复杂任务求解**：解决复杂任务的能力仍不及前沿模型，需要进一步完善基础模型和后训练方案。

### 未来方向

- 串行扩展（通过上下文管理）和并行扩展的最优组合
- 推理链的智能密度优化
- 进一步扩展预训练计算以弥补知识差距

### D. IOI、ICPC 世决赛、IMO 和 CMO 的评估方法

- **IOI**：每个问题采样 500 个候选解答，经过多阶段过滤，选择 50 个具有最长思考痕迹的样本进行最终提交。
- **ICPC**：每个问题生成 32 个候选解答，应用相同的过滤标准选择提交。
- **IMO 和 CMO**：采用生成-验证-改进循环，模型迭代改进解答直至达到完美自评估或达到最大修订次数。
